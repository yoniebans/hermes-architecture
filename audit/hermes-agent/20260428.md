## Per-page findings

### index.html
**Essence:** C4 system architecture — L1 context, L2 containers, L3 components (core engine, tool system, gateway, state), plus data flow pipelines and tech stack. The thesis: Hermes is a multi-entry-point agent with one core engine, pluggable tools, and composable backends.
**Concepts owned:** System boundary, entry points (6), AIAgent, conversation loop, tool system, gateway, state & persistence, data flows, tech stack.
**Rating:** mostly accurate

#### Accuracy

**Confirmed:**
- Version v0.11.0 matches `pyproject.toml`
- 6 entry points (CLI, TUI, Web Dashboard, Gateway, ACP, MCP) — all verified
- Conversation loop pipeline (Build Prompt → Get Tools → Call LLM → Parse → Execute → Compress) — matches `run_agent.py` structure
- Tool system architecture: `ToolRegistry` singleton, `model_tools.py` orchestration, toolset composition — accurate
- Gateway: `GatewayRunner`, `SessionStore`, `DeliveryRouter` — all exist and described accurately
- Execution backends: Local, Docker, SSH, Modal, Daytona, Singularity — all 6 verified in `tools/environments/`
- Subagent: max 3 concurrent, max depth 2, restricted toolsets — confirmed
- All refs.js paths exist in codebase ✅
- Auxiliary model system: 8 channels listed match codebase
- Max 90 iterations default — confirmed

**Stale:**
- **KPI "30+ Built-in Tools"** — `_HERMES_CORE_TOOLS` in `toolsets.py` lists 33 tool functions. The "30+" is conservative but technically still true.
- **KPI "16 Platform Adapters"** — codebase now has **20 adapter classes** (see Discovery below). The atlas lists exactly 16 tags. **Stale by 4 platforms.**
- **KPI "17 LLM Providers"** — `PROVIDER_REGISTRY` in `hermes_cli/auth.py` has **30 entries** plus OpenRouter and local Ollama. The "17" is significantly stale. The L1 diagram label says "+14" after listing 3, implying 17. Actual count is 30+.
- **KPI "7 Memory Providers"** — codebase has **8** in `plugins/memory/`: honcho, openviking, mem0, hindsight, holographic, retaindb, byterover, **supermemory** (missing from atlas).
- **State section: "Schema v6 with migrations"** — `hermes_state.py` defines `SCHEMA_VERSION = 10`. Stale by 4 schema versions.

**Misleading:**
- **"6 Exec Backends"** KPI card — there are 6 backends in concept, but `tools/environments/` has 10 `.py` files including `base.py`, `file_sync.py`, `managed_modal.py`, `modal_utils.py`. The count of "6" refers to user-facing backends (local, docker, ssh, modal, daytona, singularity) which is correct, but the directory has more files. Not misleading for a human reader, just worth noting.

**Wrong:**
- None identified.

#### Coverage

**Should cover but doesn't:**
- **RL Environments** get one dashed-line box in the L2 diagram (`rlenv`) but no L3 section. Given the `environments/` directory has benchmarks, SWE env, terminal test env, and tool call parsers, this is a meaningful omission for a system architecture page. At minimum a brief card.
- **Batch Runner** (`batch_runner.py`) — parallel batch processing entry point, not mentioned anywhere.

**Doesn't belong (discipline violations):**
- The **Tech Stack section** is borderline. It's a list of dependencies, not system structure. It doesn't help build a mental model of the *shape* — it's reference material. Could argue it belongs in CONTEXT.md. However, it does frame the technology choices at helicopter level.

#### Discovery (what the atlas missed)

- **4 new platforms since atlas was written:** Yuanbao (元宝), BlueBubbles (iMessage), Weixin (personal WeChat), QQBot. These are substantial adapters, not stubs.
- **Supermemory** — 8th memory provider, not listed.
- **Schema evolution** — 4 migrations happened (v6→v10) since the atlas was written. The schema section should at least reflect the current version.
- **`hermes-cron` toolset** — the cron system now has its own platform-style toolset (`hermes-cron` in TOOLSETS), treated as a first-class entry point. The atlas treats cron as a subsystem of the core, not an entry point.
- **WecomCallback** — separate adapter from WeCom (self-built app vs bot), meaning there are effectively two WeCom modes.
- **Total toolset count** — atlas data-model page says "22 toolset groups". Actual count from `toolsets.py` is significantly higher (30+ including all platform toolsets). The leaf/composite/platform decomposition in the data model page needs updating.

#### Structural quality

- Discipline: all prose ≤ 3 lines. Orienting, not essay-like. ✅
- Diagrams: Mermaid sources use correct `graph TD` + `classDef` patterns. ✅
- `data-ref` accuracy: all slugs used on this page resolve to valid `refs.js` entries. ✅
- Navigation: TOC matches sections. Companion-page footer links work. ✅
- One `min-height` inline style on `.mermaid-wrap` (line 144, 227, 490) — against the discipline rules in brownfield-atlas-genesis ("don't add min-height to diagram containers").

### sequence-diagrams.html
**Essence:** Complete interaction sequence diagrams tracing every message lifecycle — CLI, gateway, TUI, agent loop, compression, tool execution, subagent delegation, and error recovery.
**Concepts owned:** 7 named flows + overview. CLI flow, Gateway flow, TUI flow, Agent loop, Context compression, Tool execution, Subagent delegation, Error recovery.
**Rating:** mostly accurate

#### Accuracy

**Confirmed:**
- CLI flow participants and sequence — matches `cli.py` → `run_agent.py` path
- Gateway flow: Platform → Adapter → GatewayRunner → SessionStore → Smart Route → AIAgent → Deliver — accurate
- TUI flow: User → TUI (Ink) → GatewayClient → TUIGateway (Python) → SlashWorker/AIAgent → LLM — matches architecture
- Agent loop: AIAgent → ContextCompressor → PromptCache → LLMClient → ToolDispatch → MemoryManager — accurate
- Compression: head protection, tail budget, middle summarization, iterative re-compression — confirmed
- Subagent: parent spawns via ThreadPoolExecutor, restricted toolset, max depth/concurrent limits — confirmed
- Error recovery: credential rotation, context compression, fallback chain — accurate

**Stale:**
- **Web Dashboard flow** — missing entirely. The dashboard is a real entry point with its own message flow (browser → FastAPI → WebSocket → TUI gateway → AIAgent). It's listed as an entry point on index.html but has no sequence diagram.
- **ACP flow** — also missing. The ACP server handles VS Code/editor integration with its own session management, but no sequence diagram exists for it.

**Misleading:**
- None identified.

#### Coverage

**Should cover but doesn't:**
- **Web Dashboard message flow** — deserves its own sequence or a note on TUI flow that the dashboard shares the same path via WebSocket.
- **MCP Server flow** — how external MCP clients interact with Hermes. Listed as entry point, no sequence.
- **Cron job execution sequence** — index.html has a pipeline card for this, but no proper sequence diagram with participants exists.

**Doesn't belong:**
- Nothing flagged.

#### Discovery (what the atlas missed)

- **Batch Runner flow** — `batch_runner.py` processes multiple tasks in parallel. No sequence.
- **Plugin hook execution** — 14 lifecycle hooks fire at various points in the flows. The sequences don't show where hooks execute (e.g., `pre_tool_call` before tool execution, `post_llm_call` after LLM response). This is a cross-cutting concern worth a note.

#### Structural quality

- Discipline: prose is orienting, each flow gets a reading key. ✅
- Diagrams: all `sequenceDiagram` syntax, participants properly labeled. ✅
- `data-ref` coverage is better here than data-model (10 distinct slugs). ✅
- TOC matches sections. ✅

4. **Add RL Environments and Batch Runner.** RL has a dashed box on L2 but no substance — the `environments/` directory has benchmarks, SWE env, tool call parsers. Batch runner processes tasks in parallel. Neither appears meaningfully.

### Coherence — the atlas confuses

5. **Add cross-references between tool system descriptions.** index.html L3 section and data-model.html registry schema both describe the tool system at depth. Neither points to the other. Reader doesn't know which page owns the concept.

6. **Consolidate memory provider lists.** Listed independently on 3 pages (index, data-model, diataxis). When supermemory was added, none were updated. Single source of truth needed with cross-references from the other two.

7. **Show plugin hook execution points in sequence diagrams.** 14 lifecycle hooks fire at specific points in every flow. Someone writing a plugin can't trace where their hook fires from the current sequences.

8. **Update KPI numbers across all pages.** Platforms 16→20, providers 17→30+, schema v6→v10, memory providers 7→8, toolsets 22→30+.
